## Findings

### CRITICAL — listener leak across keep-alive socket reuse

**File:** `dashboard.mjs`, in `_forwardToAnthropicRaw`'s socket
close handler.

**Code as committed in 79389b3:**

```js
req.on('socket', (sock) => {
  sock.on('close', () => {
    _reject(new Error('upstream tcp socket closed'));
  });
});
```

**Bug:** With `keepAlive=true` (which I just enabled in the same
commit via `_upstreamAgent`), the underlying TCP/TLS socket is
returned to the agent's pool after each request and reused for the
next request. Each call to `_forwardToAnthropicRaw` adds ONE close
listener to the same socket; nothing removes them.

After ~11 reuses on the same pooled socket, Node emits
`MaxListenersExceededWarning`. More importantly, each closure
captures the request-scoped `_reject` (which references the entire
settle wrapper closure scope), so the socket pins memory
proportional to (total requests) × (pool size). Over hours of
operation this is a slow but unbounded leak — exactly the failure
profile this commit set out to FIX, in a different shape.

The closures don't break correctness (each `_reject` is a no-op
after settled), but the memory + warnings are real.

**Severity:** Critical (the audit chase was about long-uptime
leaks; introducing another long-uptime leak would defeat the
purpose).

**Fix:** Bind the close listener to a named handler and remove it
on `req.once('close', ...)`. Per Node's HTTP contract,
`request.on('close')` fires reliably on BOTH success and failure
paths, so this is a safe place to clean up.

```js
req.on('socket', (sock) => {
  const onSockClose = () => _reject(new Error('upstream tcp socket closed'));
  sock.on('close', onSockClose);
  req.once('close', () => sock.removeListener('close', onSockClose));
});
```

**Status:** ✅ Addressed in follow-up commit (this session).

### MEDIUM — `req.on('close')` fires on EVERY successful request, not just on premature close

**File:** Same function.

**Code:**
```js
req.on('close', () => {
  _reject(new Error('upstream socket closed before response'));
});
```

**Observation:** `request.on('close')` fires when the request is
fully complete — including the success path, AFTER the response is
fully consumed. In the success case the `_resolve(res)` call has
already settled the Promise, so this `_reject` is a no-op via the
settle wrapper. The error message text "upstream socket closed
before response" only ever leaves the function if it's the FIRST
settle, which only happens when the connection genuinely was torn
down before any response was received. So the message IS accurate
in every case where the user sees it.

**Status:** ✅ No fix needed — settle wrapper is correct.

### LOW — `gcAccountSlots` may purge slots whose `lastDispatchAt === 0`

**File:** `lib.mjs::gcAccountSlots`.

**Observation:** A slot created via `_slot(fp)` initializes
`lastDispatchAt = 0`. The first call to `acquireAccountPermit` sets
it to `Date.now()`. So a slot with `lastDispatchAt = 0` is ONLY one
that has been created but never dispatched (transient internal
state). Such a slot is safe to purge — it'll be recreated on next
use.

**Status:** ✅ No fix needed — purging an unused slot is correct
behavior, not a bug.
